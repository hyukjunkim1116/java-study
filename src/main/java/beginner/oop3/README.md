3.객체 지향 프로그래밍

3-1 절차 vs 객체 지향 프로그래밍
```html
절차 지향 프로그래밍은 절차를 지향. 실행 순서를 중요시. "어떻게"를 중심으로 프로그래밍
객체 지향 프로그래밍은 객체를 지향. 객체를 중요시. "무엇을" 중심으로 프로그래밍
절차 지향은 데이터와 해당 데이터에 대한 처리 방식이 분리되어 있다. 객체 지향은 데이터와 그 행동이 하나의 "객체"안에 포함되어 있다.
```

3-2 절차 지향 프로그래밍의 한계
```html
절차 지향 일때는 데이터와 기능이 분리되어서 유지보수 과점에서 관리 포인트가 늘어난다.
```

3-3 클래스와 메서드
```html
클래스는 기능 역할을 하는 메서드도 포함 가능
static 쓰면 객체 생성 안해도 메서드 호출 가능. 없으면 객체 생성해야 사용 가능
클래스는 속성(데이터,멤버 변수)와 기능(메서드) 정의 가능. 객체는 자신의 메서드를 통해 자신의 멤버 변수에 접근 가능.
객체의 메서드 내부에서 접근하는 멤버 변수는 객체 자신의 멤버 변수이다.
```

3-4 객체 지향 프로그래밍
```html
객체 지향. 어떤 속성을 가지고 어떤 기능을 제공하는지 초점 맞추자
객체를 만들어서 제공하는 개발자와 이를 사용하는 개발자가 분리되어 있다고 생각하자.
객체 사용하는 개발자의 입장에서 객체를 생성하고 메서드를 호출하기만 하면 된다. 데이터는 전혀 사용하지 않는다.
내부에 어떤 속성이 있는지 전혀 몰라도 된다. 단순히, 필요한 기능을 호출해서 사용하기만 하면 된다.
```

3-5 캡슐화
```html
속성과 기능이 하나의 캡슐에 쌓여 있는 것 같은 모습. 속성과 기능을 하나로 묶어서 필요한 기능을
메서드를 통해 외부에 제공하는 것을 캡슐화.
진짜 현실 물건을 사용하는 것 처럼 느껴진다. 코드가 더 읽기 쉽고 속성과 기능이 한 곳에 있기 때문에 변경이 더 쉬워진다
유지보수를 할때 해당 클래스 내부만 변경하면 된다. 사용하는 개발자는 코드를 변경하지 않아도 된다.
```

3-6 정리
```html
객체 지향 프로그래밍과 절차 지향 프로그래밍은 서로 대치되는 개념이 아니다.
객체 지향은 객체의 설계와 관계를 중시한다. 절차 지향은 데이터와 기능이 분리되어 있고 작동 순서에 초점을 맞춘다.
절차 지향은 데이터와 그 처리 방식이 분리되어 있다. 객체 지향은 데이터와 그 행동이 객체 안에 포함되어 있다.
객프는 모든 사물을 속성과 기능을 가진 객체로 생각하는 것이다!
이렇게 단순화 하면 세상에 있는 객체들을 컴퓨터 프로그램으로 쉽게 설계할 수 있다.
```
