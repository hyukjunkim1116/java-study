9.예외 처리

9-1 예외 처리가 필요한 이유
```html
외부 서버와 통신할 때 다양한 문제들이 발생한다.
외부 서버와 연결에 실패하거나 데이터 전송에 문제가 발생한다.
GC는 JVM 메모리 상에 있는 인스턴스를 자동으로 해제할 수 있다. 하지만 외부 연결과 같은 외부 자원은 자동으로 해제되지 않는다.
따라서 외부 자원을 사용한 후에는 연결을 해제해서 반드시 외부 자원을 반납해야한다.
```

9-2 정상 흐름과 예외 흐름
```html
정상 흐름과 예외 흐름이 섞여 있으면 코드를 이해하기 어렵다. 이 문제를 해결하기 위해 예외 처리 메커니즘을 사용한다.
```

9-3 자바의 예외 계층
```html
자바의 예외 처리는 try, catch, finally, throw, throws 키워드를 사용한다.
그리고 예외 처리용 객체를 제공한다.
예외도 Object 객체를 상속 받는다. Throwable은 최상위 예외이다.
그 아래 Error 와 Exception 객체가 있다. Error은 어플에서 복구가 불가능한 시스템 예외이므로 개발자는 이 예외를 잡으려고 하면 안된다.
Exception은 어플 로직에서 사용할 수 있는 실질적인 최상위 예외이다.
그 하위는 RuntimeException을 제외한 나머지가 컴파일러가 체크하는 체크 예외이다.
RuntimeException은 컴파일러가 체크하지 않는 언체크 예외이다. 이게 발생할때 런타임 예외라고 부른다.
```

9-4 체크 예외 vs 언체크 예외
```html
체크 예외는 개발자가 명시적으로 처리해야 한다. 언체크 예외는 명시적으로 처리하지 않아도 된다.
```

9-5 예외 기본 규칙
```html
예외는 폭탄 돌리기와 같다. 발생하면 잡아서 처리하거나 밖으로 던져야한다.
예외를 잡거나 던질때 지정한 예외 뿐만 아니라 그 예외의 자식들도 함께 처리 가능하다
예외를 처리하지 못하고 계속 던지면 main() 밖으로 나가게 되는데 시스템이 종료된다.
Exception을 상속 받으면 체크 예외가 되고 RuntimeException을 상속 받으면 언체크 예외가 된다.
체크 예외는 예외를 처리할 수 없을때 throw키워드로 예외를 필수로 선언해야 한다. 그렇지 않으면 컴파일 오류가 발생한다.
이는 개발자가 실수로 예외를 누락하지 않도록 컴파일러를 통해 문제를 잡을 수 있다.
하지만 모든 체크 예외를 잡거나 처리해야 하기 때문에 너무 번거로운 일이 된다.
언체크 예외는 예외를 잡아서 처리하지 않아도 throws 키워드를 생략할 수 있다. 말 그래도 컴파일러가 이런 예외를 체크하지 않기 때문에 언체크 예외라고 부른다.
이는 신경쓰고 싶지 않은 언체크 예외를 무시할 수 있다. 반면 개발자가 실수로 예외를 누락할 수 있다.
```
